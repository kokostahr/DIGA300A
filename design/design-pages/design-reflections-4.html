Folder Structure:

When setting up my website folder, I aimed to mirror the hierarchical information structure of my content. I wanted to ensure that everything was as organised as it could possibly be. Each section of the website, such as the Portfolio, Essay, Design and Blogs has its own dedicated folder that allowed me to separate any thematic concerns I may have encountered. Even within these sectioned folders, I still maintained a clear separation of folders and layers. For example, within the Design folder:

  • A dedicated folder for the HTML design blogs
  • A CSS folder which managed the visual layer for the html pages within the Design folder that differed, e.g. the design homepage versus the design reflection page.
  • A JavaScript folder that functioned in the same way as the CSS folder, but this time for JavaScript.

This setup follows a modular and scalable pattern. After a bit of guidance, I ensured that my naming conventions were clear and consistent, using lowercase, sometimes with hyphens, which also helped me navigate and update the site efficiently.

May I also note that, while I do think my folder structure was organised and well planned it, it did prove to be a challenge to work with, when it came to adding JavaScript interactivity, such as a dynamic navigation bar, on the files that were two folders deep. Unfortunately, despite my efforts, I could not figure out how to fix my navigation bar on the pages two folders deep . Despite my best efforts, I could not resolve the issue of the navigation bar failing to work and load correctly on those pages.

Although I was advised to use templates elements instead of writing HTML directly in my JS files, I found that this approach only added more confusion. I attempted to implement templates as suggested but they didn‚Äôt resolve my navigation issues, in fact they made things worse by introducing even more complications to my existing file structure. Eventually‚Ä¶out of honest frustration (and an existential crisis), I decided to leave the template method out and focus on a solution that at least partially worked for my website, even if it was not the most optimal one. Metadata

I included essential metadata in my HTML ‚Äòhead‚Äô to help browsers interpret my website correctly. CSS

All my styling is handled through external CSS files; I made sure to avoid in-line styling. Each content section has its own stylesheet and while there is some duplication across files (especially with base typography or layout), this allowed me to slightly tweak visual elements to match each section‚Äôs tone.

I used flex and grid systems for layout (especially my picText container which I am especially proud of) and used media quries to ensure responsiveness across device types. My stylesheets follow a variable-based approach for consistency, and I also tried to maintain flexibility for any design customisation that I had for the blog, design, portfolio, profile and essay pages.

JavaScript

I used JavaScript to introduce dynamic, reusable features, like the bottom navigation on the blog, design and essay pages.

I received feedback about ‚Äúcode duplication‚Äù and ‚Äúwriting everything in a global scope,‚Äù. Yes, some JavaScript was duplicated across content sections. Here's why the setup worked for me:

  • Each script was tailored to its section‚Äôs content, maintaining UX consistency while offering users unique data.
  • It fit my folder structure cleanly: scripts were scoped to the folders they belonged to.
  • I avoided inline scripting and used ‚ÄòDOMContentLoaded‚Äô events responsibly to encapsulate interactivity

Still, I acknowledge that the code structure could benefit from modular imports and exports, and hope to implement, using JavaScript modules in future projects‚Ä¶or iterations of this website. I also received feedback regarding my use of variable declarations. I still do get confused between each variable but do know that through practical application, I am learning to understand the importance of choosing the right declaration for each variable‚Äôs purpose. In future projects, I will stick to ‚Äòconst‚Äô for constant variables and leave ‚Äòlet‚Äô for variables that truly need it.

I used JS in my website to make interactive features like :

  • The lightbox gallery for viewing my portfolio items,
  • Filtering the different items in my portfolio
  • Navigation buttons on blog-like and essay pages
  • And a carousel on my homepage

All my scripting is also done in external JS files, and I made sure to avoid inline scripting. And I tried to clean up a lot of the comments in my code for the final submission, for clarity and readability
